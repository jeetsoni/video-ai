# Requirements Document

## Introduction

The Voice Settings Preview feature allows users to hear a short audio sample generated with their current voice settings (speed, stability, similarity boost, style) before submitting a pipeline job. Currently, users adjust voice sliders blindly with no way to audition the result. This feature adds a "Preview" button alongside the VoiceSettingsControls component that sends a short sample sentence to the ElevenLabs TTS API with the user's current slider values and selected voice, then plays the returned audio in the browser.

## Glossary

- **Preview_Button**: A UI control placed near the voice settings sliders that initiates a voice preview request.
- **Voice_Settings**: The set of four numeric parameters (speed, stability, similarityBoost, style) that control TTS output characteristics.
- **Preview_Audio**: A short audio clip generated by the ElevenLabs TTS API using the user's current Voice_Settings and selected voice.
- **Preview_Endpoint**: The backend API route that accepts a voice ID and Voice_Settings, calls the ElevenLabs TTS convert endpoint with a short sample text, and returns audio data.
- **Audio_Player**: The browser-side mechanism (HTML5 Audio API) that plays the returned Preview_Audio.
- **Sample_Text**: A short, fixed sentence used as input to the TTS API for generating Preview_Audio.
- **Cooldown_Period**: A minimum time interval between consecutive preview requests to prevent API abuse.
- **VoiceSettingsControls**: The existing React component that renders the four voice setting sliders.
- **Pipeline_Wizard**: The form-based flow for creating a new pipeline job, which includes VoiceSettingsControls.
- **DraftHero**: The quick-draft landing page component, which also includes VoiceSettingsControls.

## Requirements

### Requirement 1: Preview Audio Generation Endpoint

**User Story:** As a user, I want the backend to generate a short audio sample with my current voice settings, so that I can hear what the voice sounds like before committing to a full job.

#### Acceptance Criteria

1. WHEN the Preview_Endpoint receives a valid voice ID and Voice_Settings, THE Preview_Endpoint SHALL call the ElevenLabs TTS convert API with the provided parameters and a Sample_Text, and return the audio data as an MP3 binary response.
2. WHEN the Preview_Endpoint receives a voice ID that is empty or missing, THE Preview_Endpoint SHALL use the default voice ID configured in the ElevenLabs TTS service.
3. WHEN the Preview_Endpoint receives Voice_Settings with any value outside the defined VOICE_SETTINGS_RANGES, THE Preview_Endpoint SHALL return a 400 status code with a descriptive validation error message.
4. IF the ElevenLabs TTS API call fails, THEN THE Preview_Endpoint SHALL return a 502 status code with an error message indicating the upstream TTS service failure.
5. THE Preview_Endpoint SHALL use the ElevenLabs Flash model (eleven_flash_v2_5) to minimize latency for preview generation.
6. THE Preview_Endpoint SHALL set the Content-Type response header to "audio/mpeg" for successful responses.

### Requirement 2: Preview Button in Voice Settings UI

**User Story:** As a user, I want a Preview button next to my voice settings sliders, so that I can quickly audition my current settings.

#### Acceptance Criteria

1. THE VoiceSettingsControls component SHALL render a Preview_Button that is visible and accessible to the user.
2. WHEN the user activates the Preview_Button, THE VoiceSettingsControls component SHALL send the current Voice_Settings and selected voice ID to the Preview_Endpoint.
3. WHILE a preview request is in progress, THE Preview_Button SHALL display a loading indicator and be disabled to prevent duplicate requests.
4. WHEN Preview_Audio is successfully returned, THE Audio_Player SHALL automatically begin playback of the audio.
5. WHEN the user activates the Preview_Button while Preview_Audio is currently playing, THE Audio_Player SHALL stop the current playback and initiate a new preview request.

### Requirement 3: Audio Playback Management

**User Story:** As a user, I want smooth audio playback of the preview sample, so that I can clearly hear the effect of my voice settings.

#### Acceptance Criteria

1. THE Audio_Player SHALL play the Preview_Audio using the browser HTML5 Audio API.
2. WHEN Preview_Audio playback completes naturally, THE Audio_Player SHALL reset the Preview_Button to its idle state.
3. IF the Audio_Player fails to play the Preview_Audio, THEN THE VoiceSettingsControls component SHALL display an inline error message describing the playback failure.
4. WHEN the user navigates away from the page containing VoiceSettingsControls, THE Audio_Player SHALL stop any in-progress playback and release audio resources.

### Requirement 4: Rate Limiting and Abuse Prevention

**User Story:** As a system operator, I want preview requests to be rate-limited, so that the ElevenLabs API quota is not exhausted by excessive preview usage.

#### Acceptance Criteria

1. THE Preview_Button SHALL enforce a Cooldown_Period of 3 seconds between consecutive preview requests on the client side.
2. WHILE the Cooldown_Period is active, THE Preview_Button SHALL remain disabled and display the remaining cooldown time to the user.
3. THE Preview_Endpoint SHALL enforce a server-side rate limit of 10 preview requests per minute per client IP address.
4. IF the server-side rate limit is exceeded, THEN THE Preview_Endpoint SHALL return a 429 status code with a Retry-After header indicating when the next request is allowed.

### Requirement 5: Voice ID Propagation

**User Story:** As a user, I want the preview to use the voice I have currently selected, so that the preview accurately represents what my final video will sound like.

#### Acceptance Criteria

1. WHEN the user changes the selected voice in the VoiceSelector, THE VoiceSettingsControls component SHALL use the newly selected voice ID for subsequent preview requests.
2. THE VoiceSettingsControls component SHALL accept the selected voice ID as a prop from the parent component (DraftHero or Pipeline_Wizard).
3. WHEN no voice ID is provided to VoiceSettingsControls, THE VoiceSettingsControls component SHALL use the DEFAULT_VOICE_ID from the shared package for preview requests.

### Requirement 6: Request Validation Schema

**User Story:** As a developer, I want the preview request payload to be validated against a schema, so that malformed requests are rejected before reaching the TTS service.

#### Acceptance Criteria

1. THE Preview_Endpoint SHALL validate the request body against a Zod schema that enforces: voiceId as an optional string, and voiceSettings as an object with speed, stability, similarityBoost, and style as required numbers within their respective VOICE_SETTINGS_RANGES.
2. WHEN the request body fails schema validation, THE Preview_Endpoint SHALL return a 400 status code with the first validation error message.
3. THE validation schema SHALL be defined in the shared package alongside the existing voiceSettingsSchema.
